# pkgdown Flow (Setup -> Preview -> Publish -> Maintain)

This reference describes a recommended modern workflow for building and publishing an R package website with `pkgdown`, typically to GitHub Pages.

It is written to be practical: a sequence of steps, checks, common variants, and troubleshooting cues.

## Overview

There are two distinct contexts:

1. Local preview (developer machine)
- Build to a local output directory (commonly `docs/`).
- Iterate quickly on content and structure.

2. CI publishing (GitHub Actions)
- Rebuild the website in an automated workflow.
- Deploy the result to GitHub Pages (commonly via a `gh-pages` branch).

Key idea: use local builds for iteration, and use CI for publishing.

## Prerequisites

### Repository prerequisites

- Valid R package structure:
- `DESCRIPTION` at repo root
- `R/` source directory
- Optional but typical: `NAMESPACE` generated by roxygen2
- Documentation entry point for website home:
- `pkgdown/index.md`, or
- `index.md`, or
- `README.md`

### Tooling prerequisites

- R and required packages:
- `pkgdown`
- `usethis` (recommended)
- `devtools` (optional but common)
- If you use Quarto (`.qmd`) articles or vignettes:
- Quarto available locally (CI can install it if configured)

### If you use README.Rmd or index.Rmd

pkgdown builds the home page from markdown files. If you maintain `README.Rmd` or `index.Rmd`,
you must knit/render them to `README.md` / `index.md` yourself, and commit the generated `.md`.
pkgdown does not knit these home-page sources for you.

## Recommended Setup Flow (GitHub Pages + Actions)

### Step 1: Initialize pkgdown configuration

Recommended command:

```r
usethis::use_pkgdown_github_pages()
```

What this usually does conceptually:

- sets up pkgdown config (calls `use_pkgdown()` first)
- prepares GitHub Pages publishing (commonly with a `gh-pages` branch)
- adds a pkgdown GitHub Actions workflow
- wires site URL into config (and often `DESCRIPTION`)

Minimal manual alternative:

- create `_pkgdown.yml` manually (start from a minimal template)
- configure GitHub Pages and Actions manually

Checks:

- `_pkgdown.yml` exists at repo root
- `.github/workflows/pkgdown.yaml` exists (or equivalent)
- site URL is defined in `_pkgdown.yml` (recommended)

### Step 2: Decide home strategy (README vs index.md)

Recommended for evolving packages:

- keep `README.md` lean (quick start plus links)
- create `index.md` as a richer website landing page

Common patterns:

- growing package: `README.md` short, `index.md` richer
- small package: `README.md` can serve as both README and site home until it grows

Checks:

- at least one home source exists: `pkgdown/index.md`, `index.md`, or `README.md`
- if you maintain `README.Rmd` or `index.Rmd`, confirm the corresponding `.md` is up to date

### Step 2b (optional): Add visual identity (logo + favicons)

For a more polished site, generate favicons from your package logo:

```r
pkgdown::build_favicons()
```

Skill helper script:

```bash
Rscript scripts/setup_favicons.R
```

Notes:

- `build_favicons()` requires internet access.
- Ignore `pkgdown/` in `.Rbuildignore` to avoid `R CMD check` notes.

### Step 3: Local preview build

Run a local site build:

```r
pkgdown::build_site()
```

Optional preview:

```r
pkgdown::preview_site()
```

Checks:

- site builds without errors
- home page renders as expected
- reference pages appear
- articles list appears (if present)

### Step 4: CI build for GitHub Pages

For CI publishing, prefer:

```r
pkgdown::build_site_github_pages(new_process = FALSE, install = FALSE)
```

This function is CI-oriented:

- builds into `docs/` by default
- cleans previous output by default
- adds `.nojekyll` (and `CNAME` if needed)
- avoids package install by default

Checks:

- workflow runs on pushes to default branch
- workflow produces `docs/` site output

### Step 5: Deploy to Pages

Two common publication patterns:

#### Pattern A (recommended for pkgdown): deploy to `gh-pages`

- CI deploys built output (commonly `docs/`) to `gh-pages`.
- GitHub Pages publishes from `gh-pages` root.

Pros:

- `gh-pages` contains only website output
- default branch stays source-focused

#### Pattern B: publish from default branch `/docs`

- GitHub Pages publishes from default branch and `/docs`.

Pros:

- no separate `gh-pages` branch

Cons:

- website output lives alongside source code

Most pkgdown setups using `usethis::use_pkgdown_github_pages()` plus common Actions templates follow Pattern A.

Checks:

- GitHub Pages settings point to intended source
- site is reachable at expected URL

## Common File Roles

- `_pkgdown.yml`: site config, navbar, article grouping, template options
- `README.md`: quick start, minimal example, key links
- `index.md` or `pkgdown/index.md`: richer landing page
- `vignettes/`: article and vignette sources (`.Rmd`, `.qmd`, `.md`)
- `NEWS.md`: release notes shown as News
- `logo.svg`/`logo.png` or `man/figures/logo.*`: package logo used for visual identity
- `pkgdown/favicon/`: generated favicon assets consumed by site builds
- extra `.md` files in repo root or `.github/` (for example `ROADMAP.md`, `.github/CODE_OF_CONDUCT.md`):
  copied into the built site and converted to HTML
- `.github/workflows/pkgdown.yaml`: CI build plus deploy

## Articles, Vignettes, and Get Started

### Articles (website-first)

- use when API is changing and iteration speed matters
- often created via `usethis::use_article("name")`
- in common workflows, excluded from package build via `.Rbuildignore`

### Vignettes (ship with package)

- use when workflow is stable and offline availability matters
- higher maintenance cost

### Get started

- a vignette or article named after the package commonly becomes Get started
- keep it short, stable, and beginner-friendly
- if your package name includes a `.`, use a `-` in the vignette name (for example `pack.down` ->
  `pack-down.qmd`) to match pkgdown's Get started convention

## Suggested Verification Checklist

### Local verification

- [ ] `pkgdown::build_site()` succeeds
- [ ] home page renders expected content
- [ ] reference index and key functions appear
- [ ] articles page exists (if relevant)
- [ ] internal links are correct

### CI verification

- [ ] workflow runs on push to default branch
- [ ] `pkgdown::build_site_github_pages()` step succeeds
- [ ] deploy pushes output to `gh-pages` (if using Pattern A)
- [ ] GitHub Pages is configured correctly
- [ ] published URL matches `_pkgdown.yml`

## Troubleshooting Cues

### Symptom: `gh-pages` branch is empty

Likely causes:

- branch created but deploy has not run yet
- workflow not committed to default branch
- Actions lacks permission to push to `gh-pages`

Checks:

- `.github/workflows/pkgdown.yaml` exists on default branch
- an Actions run occurred after push
- workflow has `contents: write`

Fix:

- ensure workflow is in default branch
- ensure workflow permissions allow writes
- push a commit to trigger deploy again

### Symptom: site builds locally but fails in CI

Likely causes:

- missing system dependencies
- Quarto unavailable in CI
- package dependencies not installed in CI

Checks:

- review Actions logs for missing packages/system libs
- confirm CI installs `pkgdown` and `local::.` deps
- if using `.qmd`, confirm Quarto installation path

Fix:

- update dependency install step in workflow
- add required system libraries for runner image

### Symptom: home page is not the expected one

Likely causes:

- multiple home candidates exist
- edited README while site is using `index.md` or `pkgdown/index.md`

Checks:

- does `pkgdown/index.md` exist
- does root `index.md` exist
- which source should be authoritative

Fix:

- keep only intended home source, or align content
- for richer site plus lean README, prefer `index.md`/`pkgdown/index.md`

### Symptom: articles not showing up

Likely causes:

- no sources under `vignettes/`
- `_pkgdown.yml` article selectors do not match file stems
- Quarto/Rmd build failure for articles

Checks:

- confirm `.Rmd`/`.qmd` files exist under `vignettes/`
- compare `_pkgdown.yml` `articles:` entries with vignette stems
- build locally and inspect warnings

Fix:

- rename article files or update selector rules
- ensure Quarto is available for `.qmd`

## Maintenance Loop

- keep README lean and link to website
- add and curate articles as workflows evolve
- promote stable articles to vignettes only when needed
- maintain News for releases
- periodically review navbar and article grouping as guides grow
- after renaming/removing pages, run `scripts/clean_and_build.R` to clear stale site artifacts

## Recommended Validation Order

Before deploy/release, use checks in this order:

1. `scripts/validate_pkgdown_config.R` as a fast lint pass (`--strict` in CI when desired)
2. `scripts/check_pkgdown_builtin.R` as pkgdown-native validation (`check_pkgdown()` and `pkgdown_sitrep()`)
3. `scripts/check_tutorial_urls.R` when `tutorials:` is configured
4. `pkgdown::build_site()` locally for final visual verification
